## 故障列表

### F025 - javax.servlet 导入错误
- **时间**: 2026-06-13 00:32
- **文件**: `ResumeController.java:16`
- **错误**: `package javax.servlet does not exist`
- **根因**: Spring Boot 3.x 完成 Jakarta EE 迁移，`javax.servlet` → `jakarta.servlet`
- **修复**: `import javax.servlet.http.HttpServletResponse` → `import jakarta.servlet.http.HttpServletResponse`
- **预防规则**: **所有 Controller/Interceptor/Filter 中涉及 Servlet API 的 import 必须使用 `jakarta.*` 包，绝不能用 `javax.*`**

### F026 - InterviewController 4个编译错误
- **时间**: 2026-06-13 00:34
- **文件**: `InterviewController.java:56,57,64,66`
- **错误**:
  1. `找不到符号: 方法 setEvaluatorId()` → 实体字段是 `interviewerId`
  2. `找不到符号: 方法 setComments()` → 实体字段是 `evaluation`
  3. `找不到符号: 方法 setEvaluation()` → `InterviewArrangement` 实体无此字段（评价在 evaluation 表）
  4. `找不到符号: 变量 log` → 缺少 `@Slf4j` 注解
- **根因**: 写代码时未核对实体类字段名，凭记忆/猜测编写 setter 调用
- **修复**:
  1. `setEvaluatorId()` → `setInterviewerId()`
  2. `setComments()` → `setEvaluation()`
  3. 删除 `interview.setEvaluation(...)` ，改用 `log.info()` 记录
  4. 类上添加 `@Slf4j` 注解
- **预防规则**: **调用任何 entity 的 setter 前，必须先打开对应 Entity 文件确认字段名；绝不要凭记忆写**

### F027 - PDFBox 3.x PDDocument.load() 废弃
- **时间**: 2026-06-13 00:34
- **文件**: `ResumeInfoServiceImpl.java:144`
- **错误**: `找不到符号: 方法 load(java.io.File)`，符号: 方法 `load(java.io.File)`，位置: 类 `org.apache.pdfbox.pdmodel.PDDocument`
- **根因**: PDFBox 3.x 移除了 `PDDocument.load(File)` 静态方法，改为通过 `org.apache.pdfbox.Loader` 类加载
- **修复**: `PDDocument.load(path.toFile())` → `Loader.loadPDF(path.toFile())`
- **预防规则**: **使用第三方库时必须确认版本号对应的 API，PDFBox 3.x 用 `Loader.loadPDF()`，2.x 用 `PDDocument.load()`**

### F028 - Result.success(String) 泛型类型不匹配
- **时间**: 2026-06-13 00:41
- **文件**: `InterviewController.java:75`
- **错误**: `不兼容的类型: Result<Void> 无法转换为 Result<String>`
- **根因**: 方法返回类型声明为 `Result<String>`，但调用了 `Result.success(String message)` （该方法返回 `Result<Void>`）
- **修复**: `Result.success("暂无面试摘要")` → `Result.success("暂无面试摘要", "暂无面试摘要")` （带 data 参数的重载返回 `Result<T>`）
- **预防规则**:
  > `Result` 类的方法签名速查：
  > - `success()` → `Result<T>` (T推断)
  > - `success(T data)` → `Result<T>`
  > - `success(String message)` → `Result<Void>` ⚠️
  > - `success(String message, T data)` → `Result<T>`
  >
  > **当方法返回 `Result<String>` / `Result<具体类型>` 时，必须使用带 data 参数的 `success(message, data)` 重载！**

### F029 - Service接口缺少方法声明
- **时间**: 2026-06-13 01:03
- **文件**: `CandidateStageRecordService.java` (接口), `RecruitmentPipelineController.java:68,88`
- **错误**:
  1. `找不到符号: 方法 updateStageStatusWithAutoTransition(Long, String, String, Long)`
  2. `找不到符号: 方法 switchStage(Long, Long, Long)`
  位置均为: 变量 `candidateStageRecordService`(类型 `CandidateStageRecordService`)
- **根因**: 在 ServiceImpl 实现类中新增了两个方法（`updateStageStatusWithAutoTransition`, `switchStage`），但忘记在 Service 接口中声明。Java 中接口只暴露声明的方法，Controller 通过接口类型引用调用时编译器找不到
- **修复**: 在 `CandidateStageRecordService` 接口中添加两个方法的签名声明
- **预防规则**: **在 ServiceImpl 中新增任何 public 方法后，必须同步在对应的 Service 接口添加方法签名声明！这是 Java 面向接口编程的基本要求**

### F030 - Vue模板多余闭合标签导致构建失败+路由异常
- **时间**: 2026-06-13 01:23
- **文件**: `CandidateMyApplicationsView.vue:90`
- **错误**: `[vite:vue] Invalid end tag` (第97行报错，实际根因在第90行)
- **现象**: 点击「面试管理」菜单后页面显示「录用管理」内容（或白屏/无法加载）
- **根因**: 第90行多了一个 `</div>` 闭合标签（第89行已关闭 `.app-card`），导致Vue模板解析器标签嵌套错位，整个Vite构建失败。虽然错误文件是 CandidateMyApplicationsView.vue 而非 InterviewListView.vue，但构建失败会污染模块图，影响路由和组件的正常加载
- **修复**: 删除第90行多余的 `</div>`
- **预防规则**:
  > **每次编辑 .vue 文件的 template 部分后，必须检查标签配对！**
  > - 用IDE的"匹配括号/标签"功能验证每个 `<div>` 有且仅有一个对应 `</div>`
  > - 特别注意 v-if/v-for 块内的新增/删除标签操作，极易多删/多加闭合标签
  > - 如果出现"Invalid end tag"错误，**报错行号通常不是实际出错位置**，需要向前查找多余的闭合标签

## 代码生成检查清单 (每次写代码前必读)

- [ ] **import 检查**: 所有 servlet/web 相关 import 用 `jakarta.*` 非 `javax.*`
- [ ] **实体字段核对**: 每个 setter/getter 调用前确认 Entity 类中存在该字段
- [ ] **Result 返回值匹配**: 方法返回 `Result<XXX>` 时确保调用的 success() 返回兼容类型
- [ ] **注解完整性**: 使用了 `log` 就要有 `@Slf4j`，使用了 `@Autowired` 要确认注入方式
- [ ] **第三方库版本**: 确认 pom.xml 中依赖版本与代码中使用的 API 匹配
- [ ] **表/实体关系**: 调用某实体的 setter 前确认该字段确实属于这个实体（而非关联表）
- [ ] **Service接口同步**: 在 ServiceImpl 中新增 public 方法后，**必须**在 Service 接口添加对应的方法签名声明
- [ ] **Vue模板标签配对**: 编辑 .vue template 后检查每个 `<div>`/`<el-xx>` 标签有且仅有一个闭合标签，特别是 v-if/v-for 块内的修改
- [ ] **前后端数据格式对接**: 写前端图表/列表时先确认后端返回的JSON结构，写后端接口时先确认前端期望的数据格式
- [ ] **el-option/el-select 绑定**: 使用 `:label` 时必须核对后端实体字段名（如 `positionName` 非 `name`），不要凭记忆猜测
- [ ] **LambdaQueryWrapper 条件检查**: 使用 `update(wrapper)` 时必须确认 wrapper 有条件限制，空 wrapper = 操作全表！
- [ ] **后端参数 vs 前端参数格式**: 前端用 request body ({status:"xxx"}) 时后端必须用 `@RequestBody`，前端用 query params 时后端用 `@RequestParam`
- [ ] **Map 返回 null key 检查**: 涉及 `Map.of()` 或 `groupBy` 统计结果时，每个 key 和 value 都必须确保非 null
- [ ] **关联查询 ID 一致性**: 跨表查询时必须确保传递的 ID 是目标表的正确关联字段（如 pipeline 查询用 jobId 而非 applicationId）
